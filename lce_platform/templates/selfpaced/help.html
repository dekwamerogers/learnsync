Recommended Setup Order

Do these steps in order the first time you set up LearnSync. Getting them right first avoids re-processing data later.

First-time setup checklist

1
Configure monitored countries Go to Admin → Countries. Add every country whose learners you want to track. If you leave this blank, all countries in the CSV will be processed. Once countries are set, this filter applies to both the Activity CSV and the Enrolment CSV — set it before uploading either file.
2
Upload the Enrolment CSV Go to Admin → Enrolment CSV. This populates formal enrolment dates, activation dates, graduation status, and learner profile fields (name, country, payment status). Running this first means pace calculations start from accurate enrolment dates.
3
Upload the Activity CSV Click Upload CSV in the top bar. This is the assignment-level export from the staff portal — the main data source for all progress records, health flags, and charts.
4
Ongoing routine Upload a fresh Activity CSV each time you want updated progress data. Upload a fresh Enrolment CSV whenever enrolment or activation dates have changed. You don't need to redo countries unless your monitored scope changes.

Activity CSV Workflow

The Activity CSV is the assignment-level export from the staff portal. Every row is one learner's status on one assignment. It is the primary data source — all progress records, health statuses, and snapshot trends come from it.

Step-by-step ingestion flow

1
Click "Upload CSV" in the top bar Locate the purple Upload CSV button in the navigation bar. Select your CSV file. The system immediately parses it and shows you a preview — you are not yet writing any data.
2
Review the preview You'll see a breakdown: how many rows were found, how many programmes and courses were detected, how many new learners will be created, and how many rows were flagged (couldn't be matched to a programme). Resolve any flagged rows before confirming — learner data for flagged rows is not written until they're resolved.
3
Click "Confirm" to start ingestion The file is queued for background processing. You are returned to whatever page you were on immediately — you don't wait for it to finish.
4
Six phases run automatically Phase 1: Parse CSV. Phase 2: Update programme/course catalogue. Phase 3: Upsert learner records. Phase 4: Write assignment progress. Phase 5: Recompute health flags for every learner touched. Phase 6: Save a snapshot of every active enrolment. Watch the progress bar on the job detail page — it refreshes automatically.
5
Done — check the ingestion log Go to Admin → Ingestion Log to see the job detail: rows processed, new learners created, learners updated, rows flagged, and any warnings. If any rows couldn't be matched, a link to the review queue will appear.
Only one Activity CSV can run at a time. If you try to upload a second one while a job is still processing, you'll see a warning. Wait for the current job to finish (or cancel it), then upload the next file.

What happens to flagged rows?

Flagged rows are rows the system couldn't automatically match to a programme and course. Common reasons:

  • Unrecognised eHub class name — the class name format hasn't been seen before. Go to the flagged row detail and map it to the correct programme, or add the pattern to Admin → Pattern Registry.
  • Missing course sequence number — the CSV row doesn't have a course number. Check the Course sequence number column in the raw CSV.
  • Course doesn't exist yet — the programme was identified but that course number isn't in the catalogue. Create the course under Admin → Programmes then re-resolve the flagged row.

Resolving a flagged row triggers reprocessing — the learner's progress and health are updated as if the row had matched on the first upload.

Enrolment CSV Workflow

The Enrolment CSV is a separate export that provides formal dates and profile fields that the Activity CSV doesn't always include reliably. It is the authoritative source for enrolment dates, activation dates, graduation status, and learner profile fields.

Step-by-step flow

1
Go to Admin → Enrolment CSV → Upload Select your CSV file. The system parses it and takes you immediately to a review page — nothing is written yet.
2
Map columns on the review page The system tries to auto-detect which column contains email addresses, which contains the programme name, and which contains the enrolment date. Confirm or correct these. Also map programme names from the CSV (e.g. "ALX SE Programme") to the matching internal programme (e.g. "SE").
3
Click "Confirm" to process Runs in the background. New learners are created; existing learner profile fields are updated with the latest values from this CSV.
4
Dates and graduation status are written Enrolment date and activation date are set on each (learner, programme) pair. Graduation status and graduation date are set where applicable. These feed directly into pace calculations and health flags.
Reprocess if you mapped the wrong column: Go to the job detail page and click Reprocess. Select the correct columns and re-run without re-uploading the file.

What the Enrolment CSV writes

  • Enrolment date — the official date the learner joined the programme. This is the starting point for cohort attribution.
  • Activation date — when the learner was formally marked as active in the source system. Used in pace clock logic.
  • Graduation date & is_graduated flag — marks the learner as graduated in the system without waiting for the Activity CSV to detect it.
  • Learner profile fields — first name, last name, gender, region, country, payment status, eHub profile URL, LMS profile URL. All updated on every run. The Enrolment CSV wins on these fields.
  • Phone number — if a phone number column is present, it is written to the learner record.

Running Both CSVs at the Same Time

Can you upload the Activity CSV and the Enrolment CSV simultaneously? Yes — but there's a best practice to follow.

Short answer: Yes, they can run at the same time

The Activity CSV job and the Enrolment CSV job are independent background processes. Each one only blocks a second job of its own type — they don't block each other. So you can:

  • Start uploading the Enrolment CSV
  • While it's still processing, upload the Activity CSV
  • Both will run in parallel in separate background threads

Why the order still matters

Even though they can run simultaneously, the recommended order is Enrolment CSV first, then Activity CSV. Here's why:

  • The Activity CSV computes health flags at the end of its run. If the Enrolment CSV hasn't finished creating enrolments yet, those new roster-only enrolments won't be factored into the health rollup.
  • In practice this is only a concern if the same learner appears in both files. Activity CSV learners have eHub activity data; Enrolment CSV learners are often roster-only (not yet in eHub). These populations mostly don't overlap.
  • If a learner is in both files and the jobs run simultaneously, whichever job finishes last will win on shared profile fields (name, country, payment status). Since the Enrolment CSV is the authoritative source for profile fields, running it first and then the Activity CSV second gives the right final state.
If the same learner is in both CSVs: run the Enrolment CSV first, wait for it to complete, then upload the Activity CSV. This ensures health is computed against the full and correct set of enrolments.

What about the Pod Import CSV?

The Pod Import job (Admin → Pod Import) is also independent. It can run at the same time as either the Activity CSV or Enrolment CSV. Pod Import only writes to pod assignment tables — it doesn't touch learner health, enrolment records, or progress data.

Data Model Explained

LearnSync organises everything in a hierarchy. Understanding this hierarchy makes all the metrics make sense.

The hierarchy

Learner → Enrolment → Course Enrolment → Assignment Progress (person) (programme) (one course) (one task)

A learner who is enrolled in 3 programmes has 1 Learner record, 3 Enrolment records, and as many Course Enrolment and Assignment Progress records as their activity generates.

Learner

A unique individual, identified by their email address. One person = one Learner record, no matter how many programmes they're in.

The overall health status on the Learner record is a rollup across all their enrolments using active-wins logic — if any enrolment is Active, the learner shows as Active overall.

Enrolment

Links one learner to one programme. Stores their health status, active flags, activation date, graduation status, and whether they have eHub activity data (In eHub) or are roster-only.

One learner in 3 programmes = 3 Enrolment records. This is why Enrolments can be higher than Unique Learners.

Course Enrolment

Tracks a learner's progress within one course inside a programme. Status: Not Started, In Progress, Completed, Withdrawn. Stores the learner's pass percentage and last activity date for that specific course.

Assignment Progress

The finest level of data — one record per learner per assignment. Tracks accessed date, submitted date, and whether the assignment was passed. This is what drives all health flags.

Key dates and what they mean

Enrolment dateThe date the learner formally joined the programme. Set by the Enrolment CSV. Used for cohort attribution.
Activation dateThe date the learner was formally marked active in the source system. Set by the Enrolment CSV. Used as a pace clock reference.
First Sign of Life (FSOL)The earliest date any assignment was accessed or submitted. Detected automatically from the Activity CSV. This is the best evidence the learner has actually started engaging with content.
Effective start dateUsed for pace clocks and cohort attribution. = MAX(enrolment_date, programme.start_date). If a learner enrolled before the programme launched, their clock starts from the launch date — not their early enrolment date.
Last activity dateThe most recent date any assignment was accessed or submitted across all of a learner's courses in this programme. Drives the Inactive and Dormant flags.

Health Statuses

Every enrolment (and every learner as a whole) has a health status. It is recomputed automatically on every CSV upload and whenever you trigger Recalculate Health. You never set it manually.

Active

The learner has recent activity and no concerning flags. This is the healthy state — no action needed.

What this means in practice: the learner is submitting or accessing assignments within the threshold windows, their pass rate is acceptable, and they're not stuck on any single task.

At Risk

One or more health flags are active. The learner is still engaged (not dormant) but something needs attention.

What this means in practice: they might be inactive for 8 days (just over threshold), stuck on one assignment, or have a low pass rate. Each flag tells you the specific issue.

Dormant

No activity for longer than the dormancy threshold (default: 14 days), OR the learner has been inactive since their very first sign of life for longer than the dormancy threshold.

What this means in practice: the learner has effectively disengaged. They need re-engagement outreach before they drop out entirely.

Graduated

The learner completed all courses in the programme and is marked as graduated. No flags are evaluated for graduated learners.

What this means in practice: programme complete. They may still appear in multi-programme views if enrolled elsewhere.

Not Started

The learner has no First Sign of Life date and no assignment activity of any kind.

What this means in practice: they enrolled (or were uploaded via the Enrolment CSV) but have never touched any course content. A first-touch outreach is appropriate.

Health status priority order

When computing status, the system checks in this order — first match wins:

1. Is graduated? → GRADUATED 2. No FSOL, no activity? → NOT STARTED 3. Days inactive > dormancy threshold (14d default)? → DORMANT 4. Never-activated flag AND days since FSOL > dormancy threshold? → DORMANT 5. Any other flag active? → AT RISK 6. Otherwise → ACTIVE

Learner-level health vs enrolment-level health

Each enrolment has its own health status. The learner's overall health (shown in their profile header) is a rollup using this logic:

If ALL enrolments graduated → learner = GRADUATED If ANY enrolment is ACTIVE → learner = ACTIVE If ANY enrolment is AT_RISK → learner = AT_RISK If ANY enrolment is DORMANT → learner = DORMANT Otherwise → NOT_STARTED

Note: prerequisite programme enrolments (e.g. WALX) are excluded from this rollup.

Flags — What Each One Means

Flags are the specific signals that make an enrolment At Risk. A learner can have multiple flags simultaneously. Each flag points to a different problem and suggests a different type of intervention. Thresholds for each flag are configurable per programme under Admin → Programmes → Edit.

🔴 Inactive (default threshold: 7 days)

The learner previously had assignment activity but has not accessed or submitted anything within the inactivity threshold days.

Exact trigger: days since last_activity_date > inactivity_threshold (7 days default).

Flag detail shows: how many days they have been inactive.

Suggested action: a check-in message or call. Ask if they're having trouble, if life got in the way, or if they need encouragement to continue.

Example: A learner submitted an assignment 10 days ago and nothing since. Days inactive = 10 → Inactive flag fires (threshold = 7).

🟡 Never Activated (default threshold: 3 days)

The learner has a First Sign of Life (FSOL) date — they've logged in or accessed the platform — but has never passed or even fully engaged with any assignment. The clock starts from FSOL.

Exact trigger: has FSOL date AND days since FSOL > activation_threshold (3 days default) AND learner has NOT passed the first module of their programme.

Suggested action: early welcome outreach. They've shown up but haven't engaged with content yet. A warm nudge toward the first assignment often converts these learners.

Example: A learner logged into eHub 5 days ago but has never submitted an assignment. FSOL = 5 days ago → Never Activated flag fires (threshold = 3).

🟠 Stuck on Assignment (default threshold: 5 days)

The learner accessed a specific assignment but has not submitted it, and more than the stuck threshold days have passed since they accessed it.

Exact trigger: assignment is_accessed = True AND is_submitted = False AND days since accessed_date > stuck_threshold (5 days default).

Flag detail shows: the specific assignment name and how many days they have been stuck on it.

Suggested action: targeted academic support for that specific assignment. Ask if they understand what's being asked, if there's a technical blocker, or if they need an extension.

Example: A learner opened "Assignment 3: Market Analysis" 8 days ago but hasn't submitted it. Stuck on Assignment flag fires pointing to that specific task.

🔵 Low Pass Rate (default threshold: 70%)

The learner has submitted multiple assignments but their pass rate (passed ÷ submitted) has dropped below the pass rate threshold.

Exact trigger: submitted > 0 AND (passed / submitted × 100) < pass_rate_threshold (70% default).

Flag detail shows: the actual pass rate (e.g. "58%").

Suggested action: academic support focused on understanding where the learner is struggling. This might be a content comprehension issue or test-taking strategy.

Example: A learner submitted 5 assignments and passed 3. Pass rate = 60% → Low Pass Rate flag fires (threshold = 70%).

⏸ Stalled Between Courses (default threshold: 5 days)

The learner completed a course but has not started the next one in the sequence within the inter-course threshold days.

Exact trigger: a course has completion_date set AND the next course in sequence has no activity AND days since completion_date > inter_course_threshold (5 days default).

Flag detail shows: which course was completed and which has not been started.

Suggested action: a prompt to continue. Often learners just need a reminder that the next module is waiting. They're not struggling — they're stalled at the transition.

Example: A learner completed Course 2 on May 15 but hasn't touched Course 3 by May 21 (6 days later). Stalled Between Courses flag fires.

⛔ No Onward Progress (Stalled Progression)

The learner completed the last known course in this enrolment and has no active or recent progress in any other programme either.

Suggested action: check whether there's a next module they don't know about, or whether a graduation check is appropriate.

💳 Payment Issue

The learner's payment status is anything other than Compliant — specifically: Due Soon, Grace Period, or Overdue.

This flag is not threshold-driven. It fires the moment the payment status field is not Compliant.

A learner can be Active and still have a Payment Issue flag. Health status and payment flags are independent — a progressing learner with a payment problem will show as Active with the Payment Issue badge.

Suggested action: depends on the specific status. Due Soon → reminder. Grace Period → follow up. Overdue → escalate.

How Health is Calculated

Health computation runs once per enrolment on every Activity CSV upload, and also when you manually trigger Recalculate Health from the Admin page.

Step 1 — Gather the latest activity data

All Assignment Progress records for this (learner, programme) enrolment are loaded. The system finds:

  • The most recent date across all accessed and submitted timestamps → last_activity_date
  • Total assignments submitted and total passed → pass_rate
  • Any assignments accessed-but-not-submitted → candidates for Stuck flag
  • Course completion dates → used for Stalled Between Courses flag

Step 2 — Evaluate each flag independently

Each of the six flags is evaluated as a separate function. Multiple flags can fire simultaneously. The flags are stored as a list on the enrolment record (e.g. ["inactive", "low_pass_rate"]). They are displayed as individual badges — never collapsed into a single message.

The Payment Issue flag is added after the main health computation, layered on top of the engagement-based status.

Step 3 — Derive the health status from flags

if is_graduated → GRADUATED (no flags evaluated) elif no FSOL, no activity → NOT_STARTED elif days_inactive > dormancy_threshold → DORMANT elif never_activated AND days_since_fsol > dormancy_threshold → DORMANT elif any flag active → AT_RISK else → ACTIVE

Step 4 — Roll up to learner-level overall health

After all enrolments are computed, the learner's overall_health_status is recalculated using the active-wins rollup shown in the Health Statuses section above. This is what appears in the learner list and on the learner profile header.

Threshold Defaults

Every threshold has a system-wide default. Admins can override any threshold per programme under Admin → Programmes → Edit. Changes take effect on the next upload or Recalculate Health run — they do not alter historical snapshots.

All configurable thresholds

Activation thresholdDays after FSOL before "Never Activated" flag fires. Default: 3 days. Learners who log in but don't submit anything within 3 days are flagged.
Inactivity thresholdDays since last activity before "Inactive" flag fires. Default: 7 days.
Dormancy thresholdDays of total inactivity before health status escalates to Dormant. Default: 14 days.
Stuck assignment thresholdDays after an assignment is accessed-but-not-submitted before "Stuck on Assignment" fires. Default: 5 days.
Pass rate thresholdMinimum acceptable pass rate before "Low Pass Rate" fires. Default: 70%.
Inter-course thresholdDays between completing one course and starting the next before "Stalled Between Courses" fires. Default: 5 days.
Upload warning thresholdDays since last upload before the data-freshness warning appears on the dashboard. Default: 7 days.

Dashboard Summary Tiles — Explained

The top two rows of the home dashboard show summary tiles. Here is exactly what each number counts and what it does not count.

Unique Learners (top row, leftmost)

The count of distinct individuals who have eHub activity data in at least one non-prerequisite programme. One person enrolled in three programmes = counted once.

Why can this be higher than "Enrolments"? Unique Learners includes people enrolled in programmes that haven't started yet (upcoming programmes). Enrolments only counts started programmes. A learner enrolled in one active and one upcoming programme is counted in Unique Learners but contributes only 1 to Enrolments.

Hover over the tile's ? icon to see this explanation inline.

Enrolments (top row)

The total number of (learner, programme) pairs where the programme has already started (start_date is null or ≤ today) and the learner is not unpaid.

One learner in 3 programmes = 3 enrolments. This is why Enrolments can exceed Unique Learners.

Below the count you'll see a sub-line: Roster Only — learners who exist in the Enrolment CSV but don't yet have any eHub activity data. These are enrolled but haven't started engaging with content.

Badges Issued

Count of unique learners who have completed at least one course in a programme that has awards_credentials = True. A badge is issued per course completion, not per graduation. One learner who completed 4 courses earns 4 badges but counts as 1 learner in this tile.

This tile counts LEARNERS who received at least one badge, not the total number of badges. The total badge count appears in the detailed breakdown below.

Certificates Issued

Count of enrolments where the learner is graduated in a programme that has awards_certificate = True. One certificate per (learner, programme) graduation — a learner who graduated two programmes earns two certificates and counts twice here.

Health strip tiles (row 2)

Each tile (Dormant, At Risk, Active, Graduated, Not Started) shows the learner-level count using the active-wins rollup — not the enrolment count. A learner in two programmes who is Active in one and Dormant in another counts as Active in the tile.

Every tile is clickable — click to jump straight to the learner list filtered to that health status.

The Follow-ups Due tile shows the count of interventions you logged that have a follow-up date of today or earlier, not yet marked as resolved.

Onboarding / Onboarded (blue / green badges)

Onboarding (blue): learners currently enrolled in a prerequisite programme (e.g. WALX) who have not yet graduated from it. They are working through the foundation track.

Onboarded (green): learners who have graduated from a prerequisite programme. They've completed onboarding and can fully access their main programme.

These two counts are tracked separately because prerequisite enrolments are excluded from all the main health metrics — they run in parallel, not as part of the health rollup.

Rates & Percentages

Activation Rate

Activation Rate = Activated ÷ Total Enrolled × 100

Activated = learners who have passed the first module (first course in sequence) of their programme. Specifically: a CourseEnrolment with is_passed=True for the lowest sequence_number course.

Example: 80 learners enrolled, 52 passed module 1. Activation Rate = 52 ÷ 80 × 100 = 65%

Retention Rate

Retention Rate = Retained ÷ Activated × 100 Retained = learners who are Activated AND currently Active, At Risk, or Graduated (i.e. they passed module 1 AND are still engaged — not dormant, not not-started)
Example: 52 activated, 38 are still Active/At Risk/Graduated. Retention Rate = 38 ÷ 52 × 100 = 73%. The other 14 activated but then went Dormant.

Graduation Rate

Graduation Rate = Graduated ÷ Total Enrolled × 100

End-to-end completion rate. This includes all enrolled learners in the denominator — including those who never activated.

Example: 80 enrolled, 14 graduated. Graduation Rate = 14 ÷ 80 × 100 = 17.5%

Pass Rate (course-level)

Pass Rate = Assignments Passed ÷ Assignments Submitted × 100

Stored on each Course Enrolment. Counts only assignments submitted within that specific course — not across the entire programme.

Example: A learner submitted 5 assignments in Course 2 and passed 4. Pass Rate = 4 ÷ 5 × 100 = 80%

Delta Indicators (▲ / ▼)

What the green/red arrows mean

The small arrows and numbers next to each metric on the home dashboard show the change since the previous completed Activity CSV upload.

  • ▲ 3 means this count is 3 higher than it was in the last upload
  • ▼ 5 means this count is 5 lower than it was in the last upload
  • No arrow means no previous upload exists to compare against, or the value didn't change

Deltas are computed by comparing the current live enrolment data against the most recent EnrolmentSnapshot records from the previous job.

Delta arrows reflect the previous Activity CSV upload only. Running an Enrolment CSV does not create snapshots, so it doesn't update the delta baseline.

Pace Calculations

Pace metrics are stored on each learner's pod assignment record and recomputed whenever you click Recompute Pace on a pod's detail page.

The core formulas

current_pace = courses_completed ÷ weeks_active (courses per week) required_pace = courses_remaining ÷ weeks_remaining (courses per week) ratio = current_pace ÷ required_pace weeks_active = (today − effective_start_date) ÷ 7 weeks_remaining = (pod_target_date − today) ÷ 7

Pace start date ("the clock")

The pace clock does not simply start from enrolment date. The system picks the most meaningful starting point:

  • If the learner enrolled on or after the programme start: clock starts from enrolment date.
  • If the learner enrolled before the programme launched: clock starts from their first sign of life date (or the programme start date, whichever is later). This prevents penalising learners for days that passed before the programme even existed.
  • Pod target date: the last day of the pod's target month (e.g. for a "May 2026" pod, target = 31 May 2026).

How total courses is determined

The number of courses a learner needs to complete is taken from the programme's total_courses_for_graduation field. This is set automatically when you upload the Programme Structure CSV, and it excludes the shared onboarding module (WALX). You can verify and correct it under Admin → Programmes → Edit.

If this number is wrong for your programme (e.g. the system thinks AICE has 8 courses but you consider it 6), the required pace and projected completion will be off. Correct total_courses_for_graduation in the admin and then hit Recompute Pace on the affected pods.

Pod Pace Statuses

How status is determined

ratio = current_pace ÷ required_pace ratio > 1.05 → AHEAD ratio ≥ (1 − behind_threshold%) → ON TRACK (default threshold: 20%) ratio ≥ 0.6 → BEHIND ratio < 0.6 → SIGNIFICANTLY BEHIND required_pace is None (past deadline) → SIGNIFICANTLY BEHIND current_pace = 0 and grace > 14 days → SIGNIFICANTLY BEHIND courses_remaining = 0 → COMPLETED
AheadCurrent pace is more than 5% faster than required. The learner is comfortably ahead of schedule.
On TrackCurrent pace is within the behind threshold of required (default: within 20% slower). The learner is on pace to finish by the target month.
BehindCurrent pace is 20–40% below required. The learner is falling behind but not critically. A check-in is advisable.
Significantly BehindCurrent pace is more than 40% below required, OR the target date has already passed, OR the learner has zero completions past the grace period. Needs urgent attention — likely won't finish in this pod.
CompletedAll required courses are done (courses_remaining = 0). Pace no longer relevant.

Courses Behind — Explained

What it means

"Courses Behind" answers: how many courses should this learner have finished by now, based on the pace they need, minus how many they actually finished?

expected_by_now = required_pace × weeks_active raw_behind = max(0, expected_by_now − courses_completed) courses_not_started = total_courses − courses_completed − courses_in_progress courses_behind = min(raw_behind, courses_not_started)

The critical cap: courses not yet started

The cap ensures Courses Behind never counts a course the learner has already begun. If a learner is actively working on their last course (it's IN_PROGRESS), they can't be "behind" on any additional course — there are none. So Courses Behind = 0 for that learner, and the dashboard shows "—" instead of a misleading number.

Example: AICE has 6 courses. A learner completed 5 and is in-progress on AICE-6.
courses_not_started = 6 − 5 − 1 = 0
courses_behind = min(any number, 0) = 0 → shows "—"
Pace status may still show "Behind" if they're unlikely to finish AICE-6 by the target date — that's a time concern, not a courses-count concern.
If Courses Behind still looks too high for learners on the last module, check that total_courses_for_graduation is set correctly for that programme in Admin → Programmes → Edit. If the system thinks the programme has more courses than it does, the numbers will be off. Then hit "Recompute Pace" on the pod.

Projected completion date

projected_completion = today + (courses_remaining ÷ current_pace) weeks

courses_remaining = courses not yet COMPLETED (including any currently in-progress). So if a learner is on their last course (1 remaining to complete), and their pace is 0.5 courses/week, projected = today + 2 weeks.

Projected completion is shown in red on the pod detail page if it falls after the pod's target date.

Shows "—" if the learner has zero completions (no pace to extrapolate from).

Advanced Filter Techniques

Tips for power users: combining filters precisely, using the URL to share exact views, and understanding when to use Enrolment Health instead of overall Health.

Overall Health vs Enrolment Health — when to use which

There are two ways to filter by health status, and the difference matters when a learner is in multiple programmes:

Overall Health (Health filter in the bar): Uses the learner-level rollup. Active wins. A learner Active in AICE + Dormant in CC → shows as ACTIVE. Enrolment Health (triggered from Programme Detail page links): Checks the health of one specific programme enrolment. That same learner → shows as DORMANT when filtering CC's dormant list.

Use overall Health when: you want to find learners who need attention across your full portfolio, regardless of which programme they're struggling in.

Use Enrolment Health when: you're managing a specific programme and want to see exactly who is Dormant (or At Risk) in that programme, even if they're doing fine elsewhere.

Easiest way to use Enrolment Health: go to a Programme Detail page and click any health count (e.g. "12 Dormant"). The link pre-applies Programme + Enrolment Health so you land on exactly those 12 people. You can then add more filters (Country, Flag, etc.) on top.
URL form: ?programme=3&enrolment_health=dormant — learners who are Dormant in programme #3, even if they're Active in another programme.

Using the URL to share and bookmark views

Every filter you apply updates the URL. This makes it easy to:

  • Share a filtered view with a colleague — copy the URL and paste it in Slack/email. They'll land on the exact same view.
  • Bookmark a daily view — e.g. bookmark ?health=at_risk&flag=inactive to open it each morning.
  • Create links in reports — paste a filtered URL in a Notion doc, spreadsheet, or email to make it one-click to the right list.
The URL encodes all active filters as query parameters. Multi-select filters appear as repeated params: ?health=dormant&health=at_risk

Filter logic cheat sheet

Between two different filtersAND — learner must match both. Programme = AICE AND Health = Dormant → only AICE dormant learners.
Within one filter (multiple choices)OR — learner matches any. Health = Dormant + At Risk → dormant or at-risk learners.
Course + Course Status togetherAlways pair them — selecting a course without a status shows all learners who ever touched it (any stage). Adding status narrows to that specific stage.
Enrolment Date Range + Date BasisThe basis determines which date the range is applied to. Same date window, different basis = different learners.
Clearing a single filterClick the active (blue) filter chip below the bar to remove just that one. Or use "Clear" to reset everything.

Admin Tasks

Recalculate Health

Go to Admin → Recalculate Health. This recomputes all health flags and statuses for every active enrolment using today's date and the current threshold settings. Use this after changing thresholds or correcting programme structure — it does not create new snapshots, just updates the live health data.

Pattern Registry

Go to Admin → Pattern Registry. Maps eHub class name patterns to programmes and courses. The ingestion engine checks this first before falling back to regex detection. Add or update patterns here when new eHub class name formats appear in your CSVs. Example: adding the mapping "AICE_newformat → AICE Programme, Course 1".

Country settings

Go to Admin → Countries. Lists which countries' learner rows are processed for enrolment and progress data. Rows from unlisted countries are still scanned for new courses/programmes (catalogue discovery) but no learner data is written.

Set this before uploading any CSV. If you add a country after data has already been ingested, re-upload both CSVs to backfill those learners. Use Purge unmonitored learners to remove records that slipped in before the filter was set.

Programme thresholds

Go to Admin → Programmes → Edit for the programme you want to adjust. Each threshold field shows the current value and the system default in brackets. Leave a field blank to revert to the default.

  • Activation threshold (default 3d) — days after FSOL before Never Activated fires
  • Inactivity threshold (default 7d) — days since last activity before Inactive fires
  • Dormancy threshold (default 14d) — total inactivity days before Dormant status
  • Stuck assignment threshold (default 5d) — days accessed-but-not-submitted
  • Pass rate threshold (default 70%) — minimum pass rate
  • Inter-course threshold (default 5d) — days between course completion and next start

Managing ingestion jobs

  • Retry a failed job: go to Admin → Ingestion Log → click the failed job → Retry. Re-runs the full pipeline from scratch.
  • Cancel a running job: click Cancel on the job detail page. The job checks for cancellation between each phase and stops cleanly at the next checkpoint.
  • Delete a job: removes the job record (not the data that was already written). Only available on completed/failed/cancelled jobs.
  • Review flagged rows: click the flagged row count on a completed job. For each row you can map it to the correct programme/course or skip it.

Common Questions

Why does Unique Learners show a higher number than Enrolments?

Unique Learners includes learners enrolled in programmes that haven't started yet (upcoming programmes). Enrolments only counts started programmes. A learner in one started and one upcoming programme counts as 1 Unique Learner but only 1 Enrolment.

Why is someone showing as "Significantly Behind" on the last course with 0 courses behind?

These are separate signals. Courses Behind = 0 means they don't need to start any additional courses — they're already on the last one. Significantly Behind (pace status) means they may not finish that last course before the pod's target date. Both can be true at the same time — the learner is in the right place in the programme, they just need to complete it faster.

Why does the Activated metric not match the activation_date I see in Excel?

The Activated metric is computed live from CourseEnrolment data (first module completed with is_passed=True). The stored activation_date field comes from the staff portal CSV and reflects an older or different definition of activation. The two can diverge. The live first-module check is always the authoritative Activated metric used in every report and chart.

Can I run the Activity CSV and Enrolment CSV at the same time?

Yes — they run in separate background threads and don't block each other. Best practice: run the Enrolment CSV first to ensure enrolment dates are in place, then run the Activity CSV so health computation sees complete enrolment data. See Running Both CSVs at Once for full details.

Why does Courses Behind look wrong for a specific learner?

Check the programme's total_courses_for_graduation setting (Admin → Programmes → Edit). If this is set too high, the pace engine thinks there are more courses than the learner needs to complete, inflating the "behind" number. Set it to the correct number of content modules and hit Recompute Pace on the affected pod.

Why are some learners not appearing in the Learner List?

Check: (1) Is their country in the monitored countries list? (2) Is their payment_status 'unknown' (Unpaid)? Unpaid learners are excluded from all views unless you explicitly filter to them. (3) Are they only enrolled in prerequisite programmes? WALX-only learners are excluded from the main learner list.

How do I fix a stalled or failed ingestion job?

Go to Admin → Ingestion Log → click the failed job → click Retry. This re-runs the full pipeline. If the same rows fail again, check the flagged row queue for details. If the job shows "processing" but hasn't moved in 10+ minutes, a server restart may have killed it — it will have been automatically reset to "failed" and you can safely retry.

What does "Past Target" mean in the Required Pace column?

The pod's target date has already passed. There are no more "weeks remaining", so required pace cannot be calculated. The learner shows as Significantly Behind (the deadline passed and they're not done) with a "Past Target" label instead of a pace number.
