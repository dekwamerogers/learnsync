Recommended Setup Order

Before uploading any CSV data, a few system settings must be in place. Getting these right first avoids having to re-process data later.

First-time setup checklist

1
Configure monitored countries Go to Admin → Countries and add the countries whose learners you want to track. If you skip this, all countries in the CSV will be processed. Once set, the filter applies to both the activity CSV and the enrolment CSV — so it must be done before either upload.
2
Upload the enrolment CSV Go to Admin → Enrolment CSV and upload the programme-enrolment export. This populates formal enrolment dates, activation dates, and learner profile fields (name, country, payment status). It should run before or alongside the first activity CSV so that enrolment dates are available for pace calculations.
3
Upload the activity CSV Use the Upload CSV button in the top bar to upload the assignment-level export from the staff portal. This is the primary data source for progress, health, and charts.
4
Ongoing uploads For routine refreshes, upload a new activity CSV each time. Upload a fresh enrolment CSV whenever enrolment or activation dates have changed in the source system. Countries only need updating if your monitored scope changes.

Activity CSV Workflow

The activity CSV is an assignment-level export from the staff portal. This is the primary data source — all progress records, health statuses, and snapshots are derived from it. One row equals one learner's status on one assignment.

Step-by-step ingestion flow

1
Upload the CSV Click Upload CSV in the top bar. The file is parsed immediately. One row = one learner's status on one assignment.
2
Preview & review The system detects programmes and courses automatically. You'll see a breakdown of how many new learners, courses, and assignments will be added, plus any rows it couldn't place (flagged rows). Resolve flagged rows before confirming.
3
Confirm ingestion Click Confirm. The pipeline runs in the background through 6 phases: parse → catalogue → learners → enrolments → health → snapshots.
4
Health is recomputed Every learner's health status and flags are recalculated automatically using today's date and the latest data. No manual step needed.
5
Snapshots are stored A snapshot of every enrolment's state is saved against this upload date. Snapshots power the week-on-week progression charts in Analytics.

Flagged rows

  • Unrecognised pattern — the eHub class name didn't match any known programme. Map it via the flag detail page or the Pattern Registry.
  • Missing sequence number — the system couldn't determine which course within a programme this row belongs to. Check the CSV's Course sequence number column.
  • Unresolved course — the programme was identified but the specific course number doesn't exist in the catalogue yet. Create the course via Admin → Programmes.

Country filtering

If monitored countries are configured (Admin → Countries), only learners whose country matches are processed for progress and enrolment data. Rows from unmonitored countries are still used to discover new courses and programmes in the catalogue. Countries must be set before uploading any CSV — see the Recommended setup order above.

Enrolment CSV Workflow

The enrolment CSV is a supplementary export (separate from the activity CSV) that provides formal enrolment dates, activation dates, learner profile fields, and graduation status. It is the authoritative source for these fields — they override whatever was previously stored.

Step-by-step enrolment CSV flow

1
Go to Admin → Enrolment CSV Click Upload Enrolment CSV. The file is parsed and a review page is shown immediately — no data is written yet.
2
Map columns The system auto-detects the email, programme name, and enrolment date columns from the CSV headers. Confirm or correct them on the review page. Also map any programme names in the CSV to their matching programmes in the system (e.g. "ALX SE" → Software Engineering).
3
Confirm processing Click Confirm. The upload runs in the background. New learners found here are created; existing learner profiles are updated with the latest name, country, payment status, eHub/LMS URLs from this CSV.
4
Dates are written to enrolments Enrolment date and activation date are set (or updated) on each (learner, programme) pair. These dates feed directly into pace calculations — without them, the pace clock may start from a later or earlier date than intended.

What the enrolment CSV provides

  • Enrolment date — the formal date a learner joined a programme. Set as the selected date column at review time.
  • Activation date — read from the Activation date column if present. Used as the effective start for pace when it's later than the enrolment date.
  • Graduation date & is_graduated flag — read from Program graduation date and Is program graduated columns. Marks a learner as graduated without waiting for the activity CSV to detect it.
  • Learner profile fields — first name, last name, gender, region, country, eHub profile URL, LMS profile URL, and payment status are updated on every run. The enrolment CSV is the authoritative source for these fields.

Country filtering

The enrolment CSV also respects the monitored countries list. Rows for learners in unmonitored countries are skipped entirely — unlike the activity CSV, there is no catalogue-discovery pass for unmonitored rows here.

Re-running & purging

If you mapped the wrong column at review time, use the Reprocess button on the job detail page to re-run with a corrected column selection. To clear all enrolment upload data and start fresh, use Admin → Enrolment CSV → Purge — this removes all stored upload jobs and clears the enrolment dates that were set by those uploads.

What the Data Means

LearnSync organises data in a hierarchy: Learner → Enrolment → Course Enrolment → Assignment Progress.

Learner

A unique individual, identified by email address. A learner can be enrolled in multiple programmes simultaneously. Their overall health status is a rollup across all enrolments using active-wins logic: if any enrolment is Active, the learner is Active; if all are Graduated, the learner is Graduated; otherwise the worst non-graduated status wins.

Enrolment

Links a learner to a programme. Tracks activation date, first sign of life, graduation status, and health status at the programme level. Each enrolment has its own health status independent of others.

Course Enrolment

Tracks a learner's status within a single course inside a programme. Status can be not started, in progress, completed, or withdrawn. Stores the pass percentage and last activity date for that course.

Assignment Progress

The most granular level. One record per learner per assignment. Tracks whether the assignment was accessed, submitted, and passed, plus the dates for each action. This data directly drives health flags.

Key dates explained

  • Enrolment date — when the learner formally joined the programme.
  • Activation date — when the learner first showed intent to participate (e.g. logged in, paid, or was marked active).
  • First sign of life (FSOL) — the earliest date of any real learning activity (assignment access or submission). Used as a fallback when activation date is unavailable.
  • Effective start date — the best available start date for pace calculations: the later of the learner's enrolment date and the programme's own start date.

Health Statuses

Every enrolment and learner carries a health status. Health is recomputed on every upload and whenever you trigger Recalculate Health.

Active

The learner has recent assignment activity and is making progress. No concerning flags are raised. This is the target state for all learners.

At Risk

The learner has at least one health flag (e.g. inactive, low pass rate, stuck on an assignment). They are still engaged but need attention. Shown in the At-Risk list on the dashboard.

Dormant

The learner has had no activity for longer than the dormancy threshold (default 14 days), or has been inactive beyond the dormancy period since their first sign of life. Requires intervention before they disengage entirely.

Graduated

The learner has completed all courses in the programme and been marked as graduated in the CSV, or the system derived graduation because all courses are marked completed and the total course count is met.

Not Started

The learner is enrolled but has no first sign of life date and no assignment activity. They have never engaged with the programme content.

Flags Explained

Flags are the specific reasons an enrolment is At Risk or Dormant. A learner can have multiple flags at once. Each flag has a configurable threshold set per programme under Admin → Programmes.

Inactive

The learner previously had activity but has not accessed or submitted any assignment for more than the inactivity threshold (default: 7 days). The last activity date is tracked to identify how long they have been quiet.

Never Activated

The learner has a first sign of life date but has never accessed a single assignment. They appear to have signed up but not engaged with any content. Fires after the activation threshold days have passed since FSOL (default: 3 days).

Stuck on Assignment

The learner accessed a specific assignment but has not submitted it, and more than the stuck assignment threshold days have elapsed since they accessed it (default: 5 days). The flag detail records the assignment name and how many days they have been stuck.

Low Pass Rate

The learner has submitted multiple assignments but their pass rate is below the pass rate threshold (default: 70%). Calculated as passed / submitted × 100. A low pass rate may indicate the learner is struggling with the content.

Stalled Between Courses

The learner completed a course but has not started the next one in sequence, and more than the inter-course threshold days have passed since that completion (default: 5 days). Indicates a transition gap.

No Onward Progress (Stalled Progression)

The learner completed the last known course in this enrolment and has no active or completed course work in any other programme. They appear to have finished what's available but haven't progressed further.

Payment Issue

The learner's payment status is Due Soon, Grace Period, or Overdue. This flag is added on top of the engagement-based health status — a learner can be Active but still carry a Payment Issue flag.

How Health is Calculated

Health is a multi-step computation that runs once per enrolment on every upload.

Step 1 — Gather activity data

All AssignmentProgress records for this enrolment are loaded. The system computes the most recent activity date across all accessed and submitted dates.

Step 2 — Evaluate each flag

Each flag function runs independently and produces a boolean (or detail dict for stuck/stalled flags). Multiple flags can fire at once.

Step 3 — Derive health status

if graduated → GRADUATED elif no FSOL and no activity evidence → NOT STARTED elif days_since_last_activity > dormancy_threshold → DORMANT elif NEVER_ACTIVATED flag and days_since_fsol > dormancy_threshold → DORMANT elif any flag is active → AT RISK else → ACTIVE

Step 4 — Learner-level rollup (active-wins)

if all enrolments are GRADUATED → learner = GRADUATED elif any enrolment is ACTIVE → learner = ACTIVE elif any enrolment is AT_RISK → learner = AT_RISK elif any enrolment is DORMANT → learner = DORMANT else → NOT_STARTED

Prerequisite programme enrolments (e.g. WALX) are excluded from this rollup — they are counted separately as "onboarded".

Dashboard Metrics

The first two rows of the home dashboard show summary tiles. All figures exclude prerequisite-only programmes unless stated otherwise.

Unique Learners

The count of distinct individuals enrolled in at least one non-prerequisite programme. A learner enrolled in three programmes is counted once. Health breakdown underneath uses the active-wins rollup across all their enrolments.

Enrolments

The total number of (learner, programme) pairs in started programmes. One learner in three programmes contributes 3 to this count. The breakdown by health status is at the enrolment level — it can exceed the unique learner count.

Badges Issued

Unique learners who have completed at least one course in a programme that awards credentials (awards_credentials = True). A badge is issued per course completion, not per graduation.

Certificates Issued

The count of enrolments where the learner is graduated in a programme that awards certificates (awards_certificate = True). One certificate per (learner, programme) graduation event.

Onboarding / Onboarded badges

Onboarding (blue) — learners currently enrolled in a prerequisite programme (e.g. WALX) but not yet graduated. Onboarded (green) — learners who have graduated from a prerequisite programme. These are tracked separately because prerequisite enrolments are excluded from the main health counts.

Health strip (row 2)

Each tile in the second row (Dormant, At Risk, Active, Graduated, Follow-ups Due) is a clickable shortcut to the filtered learner list for that health state. The numbers are learner-level counts using the active-wins rollup.

Rates & Percentages

Activation rate

activation_rate = (total_learners − not_started) / total_learners × 100

Measures what proportion of enrolled learners have shown any sign of life. A learner who was never activated is excluded from the numerator.

Graduation rate

graduation_rate = graduated / (total_learners − not_started) × 100

Of learners who activated (i.e. engaged at least once), what proportion have graduated. Excludes never-started learners from the denominator.

Pass percentage (course level)

pass_pct = assignments_passed / assignments_submitted × 100

Stored on each CourseEnrolment. Counts assignments submitted within that course only. An assignment must be both submitted and passed to count in the numerator.

Deltas (▲ / ▼ indicators)

The green/red change arrows on the home dashboard compare the current upload against the previous completed upload's snapshots. They show the absolute difference in count for each metric. A missing delta means there is only one upload in the system.

Pace & Pods

Pods group learners by their target graduation month. Pace metrics measure how each learner is tracking against their pod's deadline.

Pace calculations

current_pace = courses_completed / weeks_active (courses per week) required_pace = courses_remaining / weeks_remaining (courses per week) ratio = current_pace / required_pace
  • Ahead — ratio > 1.05 (learner is 5%+ faster than required)
  • On Track — ratio ≥ (1 − behind_threshold%). Default behind threshold: 20%
  • Behind — ratio ≥ 0.6 but below on-track threshold
  • Significantly Behind — ratio < 0.6, or the target date has already passed
  • Completed — all courses done (courses_remaining = 0)

Courses behind

expected_by_now = required_pace × weeks_active courses_behind = max(0, expected_by_now − courses_completed) capped at courses_remaining

Answers "how many courses should this learner have finished by now?" minus how many they have actually finished. Capped at the number of courses still left so it never exceeds the total remaining workload.

Projected completion date

projected = today + (courses_remaining / current_pace) weeks

If a learner maintains their current pace, when will they finish? Shown as None if the learner has zero completions so far (no pace to extrapolate from).

Pace start date logic

The "clock" for pace starts at the effective start date: if a learner enrolled on or after the programme start, it's their enrolment date. If they enrolled early (before the programme launched), the clock starts at their first sign of life or the programme's start date — whichever is later. This prevents penalising learners for days before the programme existed.

Home Dashboard Charts

Health Breakdown (donut)

Shows the learner-level health distribution using the active-wins rollup. Each segment represents the proportion of unique learners in that health state across all started, non-prerequisite programmes.

Learners by Programme (bar chart)

Each bar represents one programme. The bar is stacked by health status (Graduated / Active / At Risk / Dormant / Not Started), showing the full composition of each programme's learner base. Includes onboarding (prerequisite) programmes. Hover over any column to see the total.

At-Risk & Dormant — Flag Breakdown (bar chart)

Shows how many unique learners carry each flag, filtered to only At-Risk and Dormant enrolments. A learner with multiple flags is counted once per flag. This helps identify the most common intervention trigger across the portfolio.

By Country (donut)

Distribution of the unique learner base by country of residence. Excludes learners with a blank country field. Top 4 countries are listed below the chart.

Enrolments by Week (stacked bar)

Shows how many new enrolments started each calendar week, broken down by programme. The date used is the effective start date (best of enrolment date, activation date, programme start date). Only started programmes are included.

Analytics Charts

The Analytics page has a full filter bar (date basis, date range, country, programme, health) that controls all charts simultaneously. An HTMX partial swap refreshes only the charts when a filter changes.

Health Breakdown (donut)

Learner-level health distribution, filtered to the current selection. Respects all filters including date range, so you can see the health of a specific cohort who enrolled in a particular window.

Enrolments by Programme (stacked bar)

Per-programme health breakdown at the enrolment level. Each bar shows the five health states stacked. Only non-prerequisite programmes are included here.

At-Risk & Dormant — Flags

Same as the home dashboard flag chart but respects all active analytics filters. Useful for comparing flag distributions between programmes or date ranges.

Learners by Country

Top 20 countries by unique learner count for the filtered scope. The bars are coloured distinctly so each country is easy to identify.

Portfolio Mix (donut)

Shows how many learners are enrolled in exactly one programme (single-programme) versus two or more (multi-programme). Useful for understanding cross-programme engagement.

Learner Distribution by Course

An interactive chart. Select a programme using the pill buttons at the top, then filter by health state. Each bar represents one course within that programme, showing how many learners are currently on (or completed) that course. The Badges toggle shows course-level completion counts.

Rolling Enrolment Cohorts

Groups enrolments by week of their start date (effective, enrolment date, or first activity — controlled by the Date basis selector). Each column shows how many learners from that cohort week are in each health state today. Click any column to drill into the learner list for that cohort week. A second chart excludes Not Started learners for a cleaner view of engaged cohorts.

Progression Charts

The Progression Over Time section (bottom of Analytics) uses real activity dates — not upload dates — so it shows true day-by-day learner behaviour.

Daily Active Learners

For each calendar day, counts the number of unique learners who accessed or submitted any assignment. A learner is counted once per programme per day regardless of how many assignments they touched. The X axis starts from the earliest programme start date.

Cumulative Active Learners

A running total of learners who have had any activity up to and including each day. The line for each programme is always non-decreasing. Compare programmes to see how quickly each one accumulated an engaged learner base.

Daily Course Completions

How many courses were marked completed on each individual day, grouped by programme. Spikes indicate periods of high completion activity — often around assessments or programme milestones.

Cumulative Course Completions

Running total of all course completions per programme. The slope of the line reflects the completion velocity — a steeper slope means completions are accelerating.

Quick range buttons

The filter bar includes preset buttons (4 weeks, 3 months, 6 months, 1 year, All time) that set the From/To date inputs and immediately re-filter all charts. The manual date inputs still work for exact ranges.

Interventions

Interventions are logged contact records — calls, emails, or messages sent to a learner. They create an audit trail and enable follow-up tracking.

Logging an intervention

Open a learner's profile and click Log Intervention. Record the type (call / email / other), a note, and whether a follow-up is required with a date. Bulk intervention logging is available from the learner list.

Follow-up tracking

Interventions marked as requiring follow-up appear in the Follow-ups Due tile on the home dashboard when the follow-up date is today or in the past. The tile count is the number of overdue or due-today follow-ups.

Pods

A pod is a group of learners who share the same target graduation month for a given programme. Pods make it easy to compare pace and identify who needs support before the deadline.

Creating pods

Pods are created automatically when a learner is assigned to one (via the Assign to Pod button on a learner's profile or via the pod CSV import). The pod name is derived from the programme code and target month.

Pod CSV import

Upload a Google Form-style CSV via Admin → Pod Import. The import maps learner emails to programme names and target months, then assigns or moves each learner to the appropriate pod. Learners already in a pod are moved if their target month has changed.

Recomputing pace

Use the Recompute Pace button on a pod's detail page to refresh all pace metrics. This recalculates current pace, required pace, courses behind, and projected completion date for every learner currently assigned to the pod.

Admin Tasks

Recalculate Health

Available on the Admin home page and on individual ingestion job detail pages. Recomputes health flags and statuses for all enrolments (or just one programme / one job's enrolments) using today's date and the current threshold configuration. Use this after changing thresholds or correcting data without uploading a new CSV.

Pattern Registry

Maps eHub class name patterns to programmes and courses. The ingestion engine checks this registry first before falling back to regex detection. Add or update patterns here when new eHub class name formats appear in your CSVs that the engine fails to auto-detect.

Country settings

Controls which countries' learner rows are processed for enrolment and progress data. Rows from unlisted countries are still used for catalogue discovery (new courses/programmes in the activity CSV) but no learner data is written. Set this up before uploading any CSV — if countries are added after data has been ingested, re-uploading the CSVs is required to backfill the previously skipped learners. Use Admin → Countries → Purge unmonitored learners to remove learner records that slipped in before the filter was configured.

Programme thresholds

Each programme has configurable thresholds that control when health flags fire. Accessible via Admin → Programmes → Edit. Setting a threshold to blank reverts to the system default shown in brackets.

  • Activation threshold (default 3 days) — days after FSOL before Never Activated fires
  • Inactivity threshold (default 7 days) — days since last activity before Inactive fires
  • Dormancy threshold (default 14 days) — days of inactivity before status becomes Dormant
  • Stuck assignment threshold (default 5 days) — days accessed-but-not-submitted before Stuck fires
  • Pass rate threshold (default 70%) — minimum acceptable pass rate before Low Pass Rate fires
  • Inter-course threshold (default 5 days) — days between course completion and next course start before Stalled fires
